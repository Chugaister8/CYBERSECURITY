## ВСТУП

Повсюдність мереж робить їх привабливою ціллю для зловмисників. Цей розділ охоплює виклики забезпечення безпеки мережі при різних атаках, мережевих технологіях та протоколах.

Передбачається базове розуміння стека мережевих протоколів (TCP/IP).

## ЗМІСТ

## 19.1 ЦІЛІ БЕЗПЕКИ ТА МОДЕЛІ ЗЛОВМИСНИКА
*[1731, c8][1733, c1][1734, c1][1735, c6][1732, c8]*

### 19.1.1 Цілі безпеки в мережевих системах

**Тріада CIA**: конфіденційність, цілісність, доступність. Додаткові цілі: **автентичність** (атрибуція джерела), **невідмовність** (публічна верифікація), **анонімність** (відправника/одержувача).

### 19.1.2 Моделі зловмисника

- **Противник Долева-Яо** [1230]: повний контроль над мережею; читає/блокує/дублює/модифікує будь-яке повідомлення.
- **Активний vs пасивний**: пасивний (підслуховування); активний (маніпуляція).
- **На шляху vs поза шляхом**: PITM (Person-in-the-Middle) — між двома сторонами; поза шляхом — спуфінг TCP-пакетів.
- **Позиція зловмисника**: один користувач < хибний ISP < ботмережа. Інсайдер vs аутсайдер.

## 19.2 МЕРЕЖЕВІ ЗАСТОСУНКИ
*[1731, c1][1732, c1]*

### 19.2.1 Локальні мережі (LAN)

Класична помилка: сліпа довіра до LAN. **BYOD** (Bring Your Own Device): особисті ненадійні пристрої в корпоративних мережах. Клонування MAC-адрес. Питання: контроль доступу (802.1X), моніторинг, сегментація мережі.

### 19.2.2 З'єднані мережі та Інтернет

Тrafic через кілька AS; жодна AS не повністю надійна. Маршрут-хайджекінг. VPN для безпечного з'єднання мереж. Питання: E2E конфіденційність/цілісність (TLS), безпека маршрутизації (BGP).

### 19.2.3 Шинні мережі

Промислові: Modbus, KNX. Автомобільні: CAN. Вимоги реального часу конфліктують з безпекою. Спільне середовище. Питання: ретроактивне додавання безпеки, сегментація шини.

### 19.2.4 Бездротові мережі

Широкомовне середовище → підслуховування. Немає чіткої межі. Питання: контроль доступу, запобігання підслуховуванню, захист від аналізу трафіку.

### 19.2.5 Повністю розподілені мережі (DHT та P2P)

Відсутність аутентифікації однорангових вузлів. **Eclipse-атаки**: ізоляція цільових вузлів. **Sybil-атаки**: агресивне заповнення мережі. Питання: надійне маршрутизування, аутентифікація однорангових вузлів, конфіденційність.

### 19.2.6 SDN та NFV

**SDN (Software-Defined Networking)**: відокремлює площину даних від площини управління. **NFV (Network Functions Virtualisation)**: віртуалізація мережевих функцій (брандмауери, балансувальники). Питання: нові вектори атак через контролер SDN.

## 19.3 МЕРЕЖЕВІ ПРОТОКОЛИ ТА ЇХ БЕЗПЕКА

Модель TCP/IP: рівень застосунку → транспортний → Інтернет → канальний. ISO/OSI: більш детальна, 7 рівнів.

## 19.3.1 Безпека на рівні застосунку
*[1731, c8][1734, c6,c15,c19–c22][1732, c8]*

### 19.3.1.1 Безпека електронної пошти та повідомлень

**PGP та S/MIME**: E2E конфіденційність та цілісність через гібридне шифрування (симетричний ключ зашифрований відкритим ключем одержувача) та цифровий підпис. Невідмовність. Низька юзабіліті → обмежене поширення.

Альтернатива: **TLS поверх SMTP/IMAP/POP** для hop-by-hop безпеки. Лише між вузлами, а не E2E.

### 19.3.1.2 HTTPS

HTTP + TLS = HTTPS. Детально розглядається в ГЗ 16.

### 19.3.1.3 Безпека DNS

**DNSSEC**: CA-підписані DNS-записи. Не забезпечує конфіденційність. **DNS-over-TLS (DoT)** та **DNS-over-HTTPS (DoH)**: E2E конфіденційність між клієнтом та резолвером; централізація до кількох резолверів.

**DNS cache poisoning**: PITM-зловмисник підробляє DNS-записи. **DNSSEC** захищає через цифровий підпис. **Oblivious DNS-over-HTTPS (ODoH)**: довірені проксі між клієнтами та резолверами.

**DDoS через DNS**: NXDOMAIN-атаки; ампліфікація (великі відповіді на підроблені запити). Відкриті рекурсивні резолвери; обмеження швидкості.

### 19.3.1.4 Безпека NTP

Атаки зміщення часу: застарілі сертифікати, примусове очищення кешу. Локальні NTP-сервери або NTP-реалізації з довіреними часовими алгоритмами.

### 19.3.1.5 Безпека DHT

**Eclipse-атаки**: отруєння таблиць маршрутизації. Надмірність Kademlia допомагає. **Sybil-атаки**: самовибрані ідентифікатори. Контрзаходи: обмеження за IP, криптографічні головоломки, системи репутації — але всі обмежують загальність DHT.

### 19.3.1.6 Анонімна комунікація та стійкість до цензури

**Tor**: цибулева маршрутизація через 3+ OR; цибулеве шифрування. Вхідний вузол знає клієнта; вихідний — сервер; жоден не знає обидва кінці. **Onion services**: прихована ідентичність сервера. Вразливість до аналізу трафіку (кореляція входу/виходу), навіть при повністю активному Tor.

**Цензуростійкість**: **децойна маршрутизація** (on-path роутери) [581], **domain fronting** [578].

## 19.3.2 Безпека на транспортному рівні
*[1731, c8][1733, c4,c6][1732, c8]*

### 19.3.2.1 TLS (Transport Layer Security)

TLS 1.2: погодження версії/алгоритму → обмін сертифікатами → встановлення ключів (RSA або DHKE) → перевірка handshake → передача даних.

**TLS 1.3**: 1-RTT (TLS 1.2 = 2-RTT); 0-RTT режим. Лише DHKE (немає RSA-транспорту). Формально верифікований [1719, 1757].

TLS захищає від: підслуховування, IP-спуфінгу, зміни даних, PITM-атак (за умови сертифікатів).

### 19.3.2.2 Інфраструктура відкритих ключів (PKI)

CSR → перевірка ідентичності → CA підписує сертифікат → X.509. Вразливості CA → OCSP для перевірки відкликання → **Certificate Transparency** [1758]: публічні незмінні журнали. **Мережа довіри** (PGP/GPG): децентралізована PKI.

### 19.3.2.3 Безпека TCP

**TCP reset атаки**: підбирання порядкових номерів + підроблення RST. Захист: сильна рандомізація початкового порядкового номера; відхилення RST з середнього ковзного вікна.

**SYN flooding**: виснаження ресурсів для напів-відкритих з'єднань. **SYN Cookies** [RFC4987]: HMAC-ISN, не зберігати стан до відповіді ACK.

### 19.3.2.4 Безпека UDP

UDP-протоколи без перевірки IP → вразливі до спуфінгу. **Ампліфікаційні DDoS-атаки**: IP-підроблені запити до відкритих UDP-сервісів (DNS, NTP) → великі відповіді на жертву. Захисти: перевірка адреси на рівні протоколу, обмеження швидкості, відключення ампліфікуючих сервісів [1745].

### 19.3.2.5 QUIC

Новий транспортний протокол IETF (2021) [1760]. UDP замість TCP. Інтегрує TLS 1.3 handshake; шифрує більшість заголовків. Захист від ампліфікації: перевірка адреси під час handshake, ліміт коефіцієнта 3.

## 19.3.3 Безпека на рівні Інтернет
*[1731, c8][1733, c5,c9][1734, c17][1732, c8]*

### 19.3.3.1 Безпека IPv4

**IP-спуфінг**: вільний вибір IP-адреси джерела. Захист: **egress filtering** (відкидання трафіку ззовні за межами власного простору адрес); **uRPF** (Unicast Reverse Path Forwarding).

**Фрагментаційні атаки**: Teardrop, отруєння кешу DNS. Фрагментація ускладнює виявлення ненальних шаблонів.

**VPN та IPsec**: тунельний режим (шифрує весь IP-пакет, включаючи заголовок) та транспортний режим (лише корисне навантаження). **ESP** (Encapsulating Security Payload): конфіденційність + цілісність + аутентифікація. Протокол **IKEv2** (RFC 7296) для узгодження ключів.

**NAT**: приховує внутрішні IP-адреси; неявно блокує незапитані вхідні з'єднання.

### 19.3.3.2 Безпека IPv6

128-бітний адресний простір уповільнює сканування портів. Вбудований IPsec (рекомендований, не обов'язковий). Відсутність NAT: потрібен явний stateful брандмауер. Подвійний стек: захищати обидва IPv4 та IPv6.

### 19.3.3.3 Безпека маршрутизації

**IGP (усередині AS)**: RIPv2, OSPFv2 — підтримують MD5-аутентифікацію.

**BGP (між AS)**: відсутність цілісності та аутентифікації. **Хайджекінг BGP-префіксів**: оголошення IP-простору без авторизації. **RPKI** + **ROA**: криптографічно прив'язує AS до IP-префіксів; **ROV** (Route Origin Validation). Залишкові вразливості для атак шляху. **BGPsec**: підписи на оголошення шляху; великі накладні витрати; обмежене поширення.

### 19.3.3.4 Безпека ICMP

Застарілі DoS-атаки: **Ping of Death**, **ICMP flood**, **Smurf** (широкомовне ехо → ампліфікація). Прихований канал через ICMP. Мережеве рекогносцирування. Операційний баланс: часто блокують зовнішні ICMP брандмауерами.

## 19.3.4 Безпека на канальному рівні
*[1731, c8][1733, c7][1732, c8]*

### 19.3.4.1 Контроль мережевого доступу на основі портів (IEEE 802.1X)

Аутентифікація до підключення до LAN через **EAP** (Extensible Authentication Protocol). Компоненти: **supplicant** (клієнт), **authenticator** (комутатор/AP), **сервер аутентифікації**. TLS-варіанти EAP: EAP-TLS, PEAP (уникати EAP-MD5).

### 19.3.4.2 Безпека WAN (PPP, PPPoE)

PPP-аутентифікація: **PAP** (небезпечний — відкритий текст), **CHAP** (challenge-response, стійкий до відтворення, але вразливий до офлайн-брутфорсу).

### 19.3.4.3 Атаки на комутатори Ethernet

**Переповнення CAM**: підробка випадкових MAC-адрес → флуд усіх портів. **Цілеспрямований MAC-спуфінг**: копіювання MAC-адреси цілі → перехоплення трафіку. Захист: 802.1X, ліміти на MAC-адресу за портом.

### 19.3.4.4 ARP / NDP

**ARP spooﬁng**: підроблені ARP-відповіді → переприв'язування IP до MAC → PITM, захоплення сесій. Захист: статичні таблиці IP-до-MAC, виявлення аномалій ARP. NDP для IPv6: ті ж ризики; Cryptographically Generated Addresses (CGA); стабільні адреси всередині префіксу.

### 19.3.4.5 Сегментація мережі

Розбиття на ізольовані підмережі. **VLAN** (de facto стандарт). **VLAN hopping**: switch spoofing (відповідь на протоколи транкінгу) або double tagging. Усунення: явне призначення ролей портів, відключення авто-транкінгу. **VXLAN**: до ≈16M віртуальних мереж; вразливий до введення підроблених пакетів.

### 19.3.4.6 Безпека бездротових мереж

**WEP**: 24-бітний IV → повне вичерпання за < 2 годин; атака FMS через RC4 [1783, 1784]; лінійний CRC → не виявляє зловмисних модифікацій.

**WPA**: TKIP з RC4 (для сумісності); 48-бітний IV; TMAC; унікальний ключ на пакет. Власні атаки через компроміси сумісності [1785].

**WPA2**: AES-CCMP; 4-way handshake (немає PFS). Реалізаційні вразливості виявлено [1786]; загальна методологія формально верифікована [1787, 1788].

**WPA3** (2018): замінює PSK на **SAE** (Simultaneous Authentication of Equals, на основі IETF Dragonfly); PFS. **OWE** (Opportunistic Wireless Encryption) для відкритих мереж: DHKE замість PSK.

Підприємства: **802.1X** з клієнтськими сертифікатами.

### 19.3.4.7 Безпека шинних мереж

**CAN** (Controller Area Network): без аутентифікації повідомлень; зламаний ECU може підробляти критичні сигнали; без шифрування. **AUTOSAR**: нові архітектурні принципи безпеки; погана сумісність. **Сегментація**: відокремлення критичних від некритичних компонентів. **CAN-FD**: більший розмір повідомлень → менші накладні витрати для безпекових розширень. **Modbus/TCP Security Protocol** (2018): TLS-захищені оболонки для Modbus.

## 19.4 ІНСТРУМЕНТИ БЕЗПЕКИ МЕРЕЖ
*[1731, c8][1733, c5,c8,c11,c12][1734, c23][1735, c6][1732, c8]*

### 19.4.1 Брандмауер

Фільтри трафіку на основі правил (IP-адреси джерела/призначення, протокол, порти, прапорці TCP). **Stateless** (за заголовком) vs **stateful** (за потоком).

Приклад правил: дозволити вихідний HTTP/HTTPS від внутрішніх хостів; дозволити вхідний SSH до певного хоста; дозволити встановлені з'єднання; відкидати все інше.

**Шлюз застосунку (Application Gateway/proxy)**: перевірка корисного навантаження рівня застосунку; TLS-термінація. **Шлюз рівня схеми (Circuit-level Gateway)**: TCP-реле (SOCKS). **DMZ**: зовнішньо доступні сервери (вебсервер, DNS) відокремлені від внутрішньої мережі.

### 19.4.2 Системи виявлення та запобігання вторгнень (IDS/IPS)

**IDS**: моніторинг та сповіщення. **IPS**: активне блокування/скидання.

- **Сигнатурні**: порівняння з базою відомих атак (Snort [859], Zeek [1798], Suricata [1799]).
- **Аномальні**: порівняння з профілем нормального трафіку; ML-підходи; нижча точність на практиці.
- **HIDS (Host-based)**: на окремих хостах (моніторинг трафіку + сканування вірусів).
- **NIDS (Network-based)**: на стратегічних точках.

### 19.4.3 Моніторинг безпеки мережі

- **NetFlow/IPFIX**: статистика потоків (низька вартість зберігання; для криміналістики та виявлення аномалій).
- **Мережева криміналістика**: витягування вмісту корисного навантаження (NetworkMiner, Xplico).
- **Мережеве сканування**: Nmap [1816], Zmap [1767]; перерахування хостів та сервісів.
- **IP-телескопи**: публічно маршрутизовані, але незаселені діапазони → спостереження за сканами, backscatter.
- **Приманки (Honeypots)**: навмисно вразливі системи для приваблення зловмисників.
- **Сервіси репутації**: оцінка IP/доменів за минулою поведінкою.
- **SIEM**: збирання подій з IDS, брандмауерів, системних журналів → кореляція інцидентів.

### 19.4.4 Безпека SDN та NFV

SDN: виявлення DDoS; автоматизована ізоляція заражених хостів; швидке масштабування. Нові загрози: компрометація контролера SDN → глобальне перенастроювання мережі. **DoS проти контролера** [1825]; **часовий бічний канал** SDN-комутаторів [1827].

NFV: VNF можуть бути скомпрометовані → зміна конфігурації мережі. Гіпервізорна інтроспекція; зонування безпеки [1831].

### 19.4.5 Контроль мережевого доступу

**TNC (Trusted Network Connect)**: перевірка відповідності конфігурації пристрою при підключенні до мережі (Trusted Boot + TPM + атестація). Обмеження: атестує лише початковий стан; зловмисні зміни під час виконання не виявляються.

### 19.4.6 Zero Trust мережі

Радикальна відмова від сліпої довіри до внутрішніх пристроїв. Усі пристрої ненадійні до доведення зворотного. **BeyondCorp**: ідентифікація пристроїв через Network Access Control + централізований SSO. Заявки на сервіси захищені проксі доступу. Підтримка для застарілих систем: OAuth, OpenID Connect, SAML.

### 19.4.7 Контрзаходи DoS

**Об'ємні атаки**: зупиняти трафік якомога раніше (scrubbing-сервіси, BGP FlowSpec, null routing). **Атаки на рівні застосунку**: SYN cookies, обмеження швидкості, CAPTCHA.

## 19.5 ВИСНОВОК
*[1733, c5][1735, c8,c11][1732, c6]*

### 19.5.1 Мистецтво безпечних мережевих рішень

Не існує «срібної кулі». Фундаментальні засоби: **TLS** для E2E комунікації. Для асинхронних повідомлень (електронна пошта, месенджери): **E2E безпека на рівні застосунку** (PGP/S-MIME). Зовнішні загрози: Zero Trust або централізовані брандмауери + IDS. Внутрішні загрози: 802.1X або Network Access Control. Завжди необхідний взаємодоповнюючий набір засобів безпеки.

### 19.5.2 Додаткові теми безпеки мереж

Не охоплені: безпека хмари та дата-центрів, мережі з допустимою затримкою та сенсорні мережі, мережеві приховані канали, платіжні мережі, безпека фізичного рівня (Розділ 22), безпека мережевої інфраструктури, транскордонне регулювання.
