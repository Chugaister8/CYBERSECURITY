# Розділ 15. Безпека програмного забезпечення

**Frank Piessens** — KU Leuven

---

## ВСТУП

Ця Галузь знань надає структурований огляд відомих категорій вразливостей реалізації програмного забезпечення та методів, що можна використовувати для їх запобігання, виявлення або пом'якшення їхньої експлуатації.

**Визначення:** Програмна система є безпечною, якщо вона задовольняє зазначену або мається на увазі ціль безпеки (конфіденційність, цілісність, доступність). **Відмова безпеки** — сценарій, де система не досягає своєї цілі безпеки. **Вразливість** — первопричина такої відмови.

**Вразливість реалізації** — помилка у коді, що дозволяє порушити ціль безпеки, або що уможливлює конкретні техніки атаки.

Структура розділу: Категорії (15.1), Запобігання (15.2), Виявлення (15.3), Пом'якшення (15.4).

## ЗМІСТ

## 15.1 КАТЕГОРІЇ ВРАЗЛИВОСТЕЙ
*[1447][1448][1449][1013, c17][1450]*

Вразливості реалізації — порушення часткової специфікації (контракту) підкомпонента ПЗ. При порушенні контракту система входить у стан помилки; поведінка визначається деталями реалізації, якими зловмисник може скористатися.

### 15.1.1 Вразливості управління пам'яттю

Мови C та C++ залишають відповідальність за управління пам'яттю програмісту. Поведінка програм з вразливостями управління пам'яттю є **невизначеною**.

- **Просторова вразливість**: звернення до масиву з індексом поза межами — **переповнення буфера**.
- **Часова вразливість**: звернення до пам'яті, що вже звільнена — **розіменування підвішеного покажчика**.

Техніки атак [1447]:
- **Пошкодження коду**: невалідне звернення до пам'яті змінює скомпільований код.
- **Захоплення потоку управління**: зміна вказівника коду (адреса повернення, вказівник функції) — пряме впровадження коду або атаки з повторним використанням коду (**ROP**, return-oriented programming, return-to-libc).
- **Атака лише на дані**: зміна змінних для підвищення привілеїв.
- **Витік інформації**: зчитування секретів або метаданих (адреси) для сприяння іншим атакам.

### 15.1.2 Вразливості генерації структурованого виводу

Програми динамічно конструюють структурований вивід (SQL, HTML, shell-команди), що споживається іншою програмою. Небезпечна практика — конкатенація рядків, куди входять ненадійні вхідні дані.

Типи **ін'єкційних вразливостей**:
- **SQL-ін'єкція**: вивід — SQL-код; широко поширена у серверних вебзастосунках.
- **Ін'єкція команд**: вивід — команда shell.
- **Ін'єкція скриптів (XSS)**: вивід — JavaScript для браузера.

Ускладнюючі чинники: підмови (HTML включає JS, CSS, SVG); **збережені ін'єкції** (вихід одної фази стає вхідними даними для наступної — Stored XSS, higher-order SQL injection).

### 15.1.3 Вразливості стану гонитви

**Стан гонитви (race condition)**: поведінка програми залежить від того, який паралельний актор першим отримує доступ до ресурсу.

**TOCTOU (Time Of Check Time Of Use)**: зловмисник може зробити перевірену умову недійсною між перевіркою та використанням. Зустрічається в: гонитвах на файловій системі (привілейовані програми), гонитвах стану сесії у вебзастосунках.

### 15.1.4 Вразливості API

Майже все ПЗ програмується проти API. Порушення контракту API призводить до стану помилки. Особливо чутливі: API до бібліотек безпеки (криптографія, контроль доступу). Є вагомі емпіричні свідчення [1452] того, що розробники часто роблять помилки у використанні криптографічних API.

### 15.1.5 Вразливості бічних каналів

**Бічний канал** — інформаційний канал, що передає інформацію про виконання програми через ефекти, від яких код програми абстрагується (час, споживання потужності, ЕМ-випромінювання, поведінка кешу).

**Приховані канали (covert channels)**: зловмисник також контролює програму, що витікає через бічний канал — цілеспрямована ексфільтрація.

**Атака Rowhammer**: зловмисно сконструйовані паттерни доступу до пам'яті викликають небажану взаємодію між комірками DRAM щільної пам'яті → перевертання бітів пам'яті.

**Атаки ін'єкції збоїв**: фізичні (перепади напруги, ЕМ-випромінювання, температура) або програмні засоби для порушення апаратури.

### 15.1.6 Обговорення

**Кращий зв'язок з цілями безпеки потребує складніших специфікацій.** Вразливості витоку інформації — порушення **специфікації інформаційного потоку (confidentiality-oriented)**; потребують порівняння двох виконань.

**Вразливості бічних каналів** за визначенням не є порушеннями специфікації на рівні вихідного коду. Загальноприйнятих контрзаходів не існує.

**Вразливості як помилки**: можуть розглядатися як варіант поняття **помилок** (faults) у надійних обчисленнях [1074].

## 15.2 ЗАПОБІГАННЯ ВРАЗЛИВОСТЯМ
*[1455, 1456, 1457][1458, c3]*

Найефективніші підходи усувають категорії вразливостей завдяки дизайну мови програмування або API. Ключова відмінність: **захоплені помилки** (trapped errors) — негайне завершення; **незахоплені помилки** (untrapped errors) — небезпечні, оскільки поведінка потім може бути довільною.

### 15.2.1 Дизайн мови та системи типів

#### 15.2.1.1 Вразливості управління пам'яттю

**Безпечна по пам'яті мова**: мова, визначення якої означає відсутність незахоплених помилок управління пам'яттю. C/C++ — небезпечні по пам'яті. Java/C# — безпечні (поєднання статичних/динамічних перевірок та збирання сміття). **Rust** досягає безпеки пам'яті без збирача сміття через систему типів з **правом власності** (ownership type system).

#### 15.2.1.2 Вразливості генерації структурованого виводу

Типова система для XML: регулярно-виразні типи [1460] гарантують, що типово-коректна програма генерує XML відповідної структури. **LINQ** (C#): синтаксис виразів запитів — явна структура запиту.

#### 15.2.1.3 Вразливості стану гонитви

**Режим власності (ownership regime)** пом'якшує проблеми псевдонімів (aliasing). Rust — перша мейнстримова мова з системою типів на основі власності.

#### 15.2.1.4 Інші вразливості

Підходи до безпеки інформаційного потоку на основі мов [1297]; SPARK — реальна мова з аналізом інформаційного потоку у компіляторі.

### 15.2.2 Дизайн API

Там, де мова програмування не запобігає вразливостям, використання API вищого рівня може знизити їх ймовірність:

- **Жирні/розумні покажчики (fat/smart pointers)** для C/C++; збирач сміття; режим власності.
- **API підготовлених виразів (Prepared Statement APIs)** — розділяють структуру SQL-запиту від вхідних даних.
- **Бібліотеки безпечної криптографії** — спрощення, безпечні значення за замовчуванням, краща документація.

**Дизайн за контрактом**: явні передумови та постумови в API; захисне програмування перевіряє їх.

**API об'єктних можливостей** [1461] підтримують мінімальні привілеї.

### 15.2.3 Практики кодування

Безпечні практики кодування суттєво знижують ймовірність вразливостей. Стандарти: **SEI CERT** (для C [1457] та Java [1462]), **MISRA** (критичні системи у C) [1463], **SPARK** (підмножина Ada з верифікацією).

Правила: уникати небезпечних функцій API; уникати невизначеної поведінки; не зберігати секрети в Java Strings; виключати вхідні дані користувача з рядків формату; підтримувати режим власності (**RAII, move semantics, smart pointers** у C++).

## 15.3 ВИЯВЛЕННЯ ВРАЗЛИВОСТЕЙ
*[1449, 1464][1458, c4]*

Компроміс між **звукністю** (soundness, відсутність помилкових негативів) та **повнотою** (completeness, відсутність помилкових позитивів). Теорема Райса: неможливо мати і те, і інше для нетривіальних категорій.

### 15.3.1 Статичне виявлення

#### 15.3.1.1 Евристичне статичне виявлення

Детектує порушення правил кодування. Семантична модель програми (AST, потік даних/управління). **Аналіз заражень (taint analysis)**: чи можуть значення з вхідних даних програми впливати на небезпечні операції; **аналіз потоку інформації** включає неявні потоки через потік управління.

Виклик: конфігурування джерел, стоків та санітайзерів.

#### 15.3.1.2 Звукна статична верифікація

Мета — звукність для добре визначених категорій. Три основні техніки:
- **Верифікація програм**: програмна логіка (логіка розділення [1332] для безпеки пам'яті та гонитв); зазвичай потребує анотацій.
- **Абстрактна інтерпретація**: автоматична; відображення значень часу виконання на абстрактні домени; ефективна для програм без динамічного розподілу.
- **Перевірка моделей**: вичерпний пошук станів; обмежена через вибух стану; **обмежена перевірка моделей** (bounded model checking) — не звукна, але знаходить багато вразливостей.

«Soundiness Manifesto» [1465] описує інженерні компроміси у практичних реалізаціях.

### 15.3.2 Динамічне виявлення

#### 15.3.2.1 Моніторинг

- **Вразливості управління пам'яттю**: динамічні монітори можливі, але дорогі.
- **Вразливості генерації структурованого виводу**: монітор через тонкозернистий динамічний аналіз заражень [1464].
- **API-вразливості**: контракти (передумови/постумови), скомпільовані у код для тестування.
- **Стан гонитви**: моніторинг дисципліни блокування.

#### 15.3.2.2 Генерація релевантних виконань

**Фаз-тестування (fuzz testing)**:
- **Чорна скринька**: лише поведінка вводу/виводу — випадкове тестування, тестування на основі моделей, мутаційний фазинг.
- **Біла скринька**: **Динамічне символічне виконання** — виконання з конкретними вхідними даними + побудова умов шляху → вирішення для нових виконань.

## 15.4 ПОМ'ЯКШЕННЯ ЕКСПЛУАТАЦІЇ ВРАЗЛИВОСТЕЙ
*[1447, 1466]*

Техніки пом'якшення реалізуються в інфраструктурі виконання (апаратура, ОС, завантажувач, VM) або вбудовуються у виконуваний файл компілятором.

### 15.4.1 Виявлення атак під час виконання

- **Стекові канарейки (stack canaries)**: виявляють порушення цілісності активаційних записів.
- **NX (No eXecute)**: виявляє спроби направити лічильник програм у пам'ять даних → виявляє атаки прямого впровадження коду.
- **CFI (Control-Flow Integrity)**: відстежує дотримання потоком управління очікуваного потоку → виявляє атаки з повторним використанням коду.

### 15.4.2 Автоматизована диверсифікація ПЗ

Диверсифікація деталей реалізації ускладнює підготовку та тестування атак.

**ASLR (Address Space Layout Randomization)**: рандомізація розташування коду, стека та/або купи.

**Тонкозерниста рандомізація** [1466]: рандомізація розташування окремих функцій, базових блоків, розподілу регістрів.

### 15.4.3 Обмеження привілеїв

**Пісочниця (sandboxing)**: виконання у контрольованому середовищі з обмеженим доступом до ресурсів. Реалізації: VM-монітори, ОС-рівнева пісочниця, Java Runtime Environment [1467]. Також пом'якшує наслідки успішного експлойта.

**Компартменталізація (compartmentalisation)**: ПЗ розподілено на відсіки з обмеженими привілеями. Приклад: компартменталізований браузер — рендерний рушій без доступу до файлової системи. **Об'єктно-орієнтовані системи можливостей** [1461] для дуже тонкозернистої компартменталізації.

### 15.4.4 Перевірка цілісності ПЗ

**Довірене завантаження (Trusted Boot)**: вимірювання акумулюються для кожної виконуваної програми; будь-яка модифікація призводить до іншого вимірювання. Доступ до секретних ключів — лише зі стану з вказаним вимірюванням [1468].
