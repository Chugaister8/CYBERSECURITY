## 16.1 ВСТУП

Веб та мобільна безпека є основними засобами взаємодії багатьох користувачів з Інтернетом. Дві ключові тенденції: **аплікація (appification)** — перехід від браузерної платформи до мобільних застосунків з вузькою функціональністю; **вебізація (webification)** — побудова застосунків на основі вебтехнологій (HTTP, HTML, JavaScript).

Обидві тенденції призвели до значного зміщення поверхні атаки на бік клієнта. Ця ГЗ не охоплює загальні вразливості ПЗ (Розділ 15), безпеку ОС (Розділ 11) та мобільні мережі (Розділ 22).

## 16.2 ФУНДАМЕНТАЛЬНІ КОНЦЕПЦІЇ ТА ПІДХОДИ
*[1474, 1475, 1476, 1477, 1478, 1479, 1480, 1481, 1482]*

### 16.2.1 Аплікація

**Аплікація** — перехід від браузерного доступу до мобільних застосунків зі спеціалізованим набором функцій. «Є застосунок для всього» — девіз аплікованих екосистем.

**Розробники-аматори (citizen developers)**: низький бар'єр входу через генератори онлайн-застосунків (OAG). Дослідження [1484] показують, що згенеровані застосунки мають тенденцію бути вразливими до атак перенастроювання та ін'єкцій коду.

### 16.2.2 Вебізація

Застосунки побудовані на вебтехнологіях: серверний Python/Ruby/Java/JavaScript та клієнтський JavaScript.

#### 16.2.2.1 URL (Uniform Resource Locators)

Синтаксис абсолютного URL: `scheme://credentials@host:port/resourcepath?query_parameters#fragment`. Кожен сегмент має специфічне значення (схема, хост, порт, шлях, параметри запиту, якір).

#### 16.2.2.2 HTTP (Hypertext Transfer Protocol)

Текстовий протокол поверх TCP/IP. HTTP-запит: перший рядок з версією, заголовки (User-Agent, Host, Accept, Content-Length, Cookie), тіло. HTTP-відповідь: версія, статус-код, заголовки (Set-Cookie), тіло.

**Cookie**: розширення HTTP. Сервер встановлює через `Set-Cookie: name=value`, клієнт повертає через `Cookie: name=value`. Широко використовуються для управління сесіями та автентифікації.

**WebSocket** [1487]: двонаправлене з'єднання поверх HTTP. Починається зі звичайного HTTP-запиту з заголовком `Upgrade: WebSocket`.

#### 16.2.2.3 HTML (Hypertext Markup Language)

Ієрархічна деревоподібна структура тегів та текстових вузлів. **DOM** визначає логічну структуру документа. Деякі символи потребують **HTML-кодування сутностей** (entity encoding). Невідповідне кодування призводить до XSS.

#### 16.2.2.4 CSS (Cascading Style Sheets)

Опис зовнішнього вигляду HTML-документів. Значення CSS можуть бути контрольовані користувачем.

#### 16.2.2.5 JavaScript

Об'єктно-орієнтована мова для браузера та сервера (Node.js). Клієнтський JavaScript не підтримує I/O безпосередньо — лише через browser API. Кожен HTML-документ отримує власний контекст виконання JavaScript; контексти суворо ізольовані.

#### 16.2.2.6 WebAssembly

Ефективний бінарний формат інструкцій для стекової VM. Пісочниця, SOP, доступ до JS у тому ж контейнері.

#### 16.2.2.7 WebViews

Дозволяють інтеграцію вебконтенту у мобільні застосунки. Двосторонній зв'язок між нативним кодом застосунку та JavaScript. **Атаки app-to-web**: ін'єкція JavaScript у WebView. **Атаки web-to-app**: впровадження ненадійного вебконтенту та використання JavaScript-містка до застосунку.

### 16.2.3 Магазини застосунків

Централізовані платформи розповсюдження ПЗ (Chrome Web Store, Apple App Store, Google Play). Перед публікацією більшість магазинів проводять перевірку безпеки — статичний та динамічний аналіз бінарних файлів. Підписання застосунків у Android: самопідписані сертифікати; оновлення — тим самим ключем. iOS: підписання Apple; непідписані застосунки не встановлюються.

### 16.2.4 Пісочниця (Sandboxing)

Ізоляція на рівні ОС-процесів (Chrome, Android).

#### 16.2.4.1 Ізоляція застосунків

У мобільних платформах кожен застосунок — у власному пісочниці та власному файловому сховищі. В Android: UID/GID, контексти безпеки на рівні ядра.

#### 16.2.4.2 Ізоляція контенту

**Політика одного джерела (SOP)** [1503]: два контексти виконання JavaScript можуть маніпулювати DOM тільки при точному збігу протоколу, DNS-імені та номера порту (**origin = protocol + DNS + port**).

Вразливість SOP: спирається на DNS, не на IP-адреси.

Сучасні браузери додають другий рубіж захисту: ізоляція сайтів (рендер у власному процесі-пісочниці).

**CSP (Content Security Policy)** [1505]: HTTP-заголовок або мета-тег, що обмежує довірені джерела скриптів та медіа. Захист від XSS.

Приклад заголовка CSP: `Content-Security-Policy: default-src 'self'; img-src *; media-src trusted-media.com; script-src trusted-scripts.com`

### 16.2.5 Контроль доступу на основі діалогів дозволів

Мобільні та веб-платформи розширюють традиційні системи, де застосунки є принципалами.

**Рівні дозволів (Android)**: нормальні (надаються мовчки) та небезпечні (потрібне схвалення користувача). Модерні Android (≥6.0): дозволи запитуються під час виконання при першому зверненні.

Проблеми: розробники запитують більше дозволів, ніж потрібно [1479]; користувачі часто ігнорують діалоги через звикання [1393].

### 16.2.6 PKI Вебу та HTTPS

HTTPS накладає HTTP поверх TLS для серверної автентифікації та захисту даних у транзиті. Обидва браузери та мобільні застосунки автентифікують HTTPS-сервери через сертифікати X.509, підписані одним зі встановлених CA (сотні СА у списку).

Попередження браузера при помилці перевірки сертифіката. HSTS [1508]: HTTP Strict Transport Security — примусове з'єднання через HTTPS.

**Шахрайські CA та Certificate Transparency**: DigiNotar (2011) [1510] — видав шахрайські сертифікати, збанкрутував. **Certificate Transparency** [1511]: незмінний журнал видачі сертифікатів CA; клієнти мають приймати лише сертифікати зі знаком підтвердження прозорості.

### 16.2.7 Автентифікація

#### 16.2.7.1 HTTP-автентифікація

**Базова HTTP-автентифікація** [1512]: сервер надсилає `HTTP 401`; клієнт кодує облікові дані Base64 та повертає в заголовку Authorization. **Небезпечна без HTTPS** (Base64 — не шифрування).

**Форм-базована HTTP-автентифікація**: HTML-форма для введення облікових даних. Також небезпечна без HTTPS.

#### 16.2.7.2 Автентифікація мобільного пристрою

Паролі, PIN-коди, паттерни (Android 3×3-сітка), відбитки пальців, розпізнавання обличчя (через TrustZone).

Паттерни розблокування мають низьку ентропію (початок верхній лівий кут, прямі лінії) [367]. Також вразливі до shoulder surfing.

### 16.2.8 Cookies

HTTP-cookie: стан сесії на клієнті. Сесійні cookie — до закриття браузера; постійні — до закінчення терміну дії. Вразливість: **перехоплення ідентифікатора сесії** [1516].

### 16.2.9 Паролі та альтернативи

Паролі — домінуючий метод автентифікації. Проблеми: слабкі паролі, повторне використання. Поточні настанови (NIST) [1517, 1518]: не вимагати складних паролів, вимагати довжину.

Контрзаходи провайдерів:

- **Політики паролів**: правила довжини та складності, чорні списки.
- **Лічильники сили паролів (PSM)**: візуальний зворотний зв'язок.
- **Менеджери паролів**: генерація, зберігання, отримання — але обмежений вплив через проблеми зручності [1523].
- **Багатофакторна автентифікація (MFA/2FA)**: PSD2 вимагає 2FA для онлайн-платежів.
- **WebAuthn** [1525]: компонент FIDO2; стандартизований інтерфейс автентифікації через криптографію з відкритим ключем.
- **OAuth**: делегована авторизація через токени доступу.

### 16.2.10 Часті оновлення ПЗ

Chrome та Firefox: автоматичні оновлення. Мобільні платформи: автоматичні оновлення сторонніх застосунків (переважно по WiFi). Проблема: застарілі сторонні бібліотеки — значна кількість розробників використовують застарілі бібліотеки [1530, 1531].

## 16.3 ВРАЗЛИВОСТІ ТА ПОМ'ЯКШЕННЯ НА СТОРОНІ КЛІЄНТА
*[1532-1544]*

### 16.3.1 Фішинг та клікджекінг

#### 16.3.1.1 Фішинг

**Фішинг** — шахрайські атаки для крадіжки облікових даних, номерів кредитних карток [1534]. Техніки: підроблені листи, вебсайти, SMS/повідомлення. Використання: орфографічні помилки в URL, піддомени, **IDN-атаки гомографів** (заміна схожих символів різних кодувань, наприклад, латинська «a» → кирилична «а»).

Атаки особливо важкі для виявлення в мобільних браузерах (адресний рядок не видно). Контрзаходи: навчання [1533], чорні списки, візуальні індикатори ТЛД.

**Атаки drive-by download**: відвідування шкідливого сайту → завантаження та встановлення шкідливого ПЗ.

#### 16.3.1.2 Клікджекінг

**Клікджекінг** — маніпуляція зовнішнім виглядом для спонукання натиснути на підроблений елемент [1535]. Прозорий/непрозорий шар поверх оригінального сайту через `iFrame`. Жертви клацають на оригінальному елементі, вважаючи, що клацнули на видимому.

Контрзаходи: відключення JavaScript/iFrame (порушує легітимні сайти); **FrameBusting** (JavaScript для виявлення iFrame, ненадійний); **X-FRAME-OPTIONS: DENY** (рекомендований серверний захист).

### 16.3.2 Сховище на стороні клієнта

#### 16.3.2.1 У браузері

**WebStorage** [1539]: пари ключ-значення. **IndexedDB** [1540]: noSQL-подібна БД. Основна проблема: зловмисники можуть маніпулювати збереженою інформацією.

Рекомендації: **криптографічний підпис** чутливих даних; **sessionStorage** для сесійних даних; очищення після сесії.

#### 16.3.2.2 У мобільних застосунках

Підписувати та шифрувати чутливу інформацію; правильна санітизація вводу; захищені механізми доступу для внутрішнього сховища. SD-карти — загальне сховище без безпечного контролю доступу.

### 16.3.3 Фізичні атаки

#### 16.3.3.1 Атаки відбитків пальців (smudge attacks)

Відновлення паролів/PIN-кодів/паттернів розблокування за жировими слідами на сенсорному екрані [1543].

#### 16.3.3.2 Підглядання через плече (shoulder surfing)

Пряме спостереження або камери/телескопи [1544]. Особливо небезпечне у громадських місцях.

## 16.4 ВРАЗЛИВОСТІ ТА ПОМ'ЯКШЕННЯ НА СТОРОНІ СЕРВЕРА
*[1550-1559]*

### 16.4.1 Ін'єкційні вразливості

Атаки ін'єкцій виникають при недостатній перевірці вхідних даних.

#### 16.4.1.1 SQL-ін'єкція

**SQL-ін'єкція** [1550, 1551] — маніпуляція запитами до реляційних БД. Приклад: `' OR '1'='1` у полі вводу розширює WHERE-умову на всі записи.

Типи: пряма (результати видимі), **сліпа SQL-ін'єкція** (результати невидимі, але можна виявити через true/false відповідь), **ін'єкція другого порядку** (зловмисний вміст зберігається для пізнішого використання).

Пом'якшення [1552, 1553]: **підготовлені вирази (Prepared Statements)** — розділяють структуру запиту від вхідних даних; ORM (Object Relational Mapping); екранування (error-prone, не рекомендується).

#### 16.4.1.2 Ін'єкція команд

Виконання довільних системних команд через вразливий застосунок [1560]. Пом'якшення: безпечна конструкція рядків команд; мінімальні привілеї; моніторинг CVE [1554].

#### 16.4.1.3 Завантажені користувачем файли

Шкідливі файли можуть запускати небажане виконання коду (наприклад, .php-файл, що виконується сервером). Пом'якшення: обмеження та фільтрація метаданих та вмісту файлів; заголовок `Content-Disposition`; обслуговування файлів з іншого домену.

#### 16.4.1.4 Включення локальних файлів (LFI)

Маніпуляція шляхами файлів для зчитування чутливих локальних файлів (наприклад, /etc/shadow). Пом'якшення: санітизація шляхів файлів; мінімальні привілеї.

#### 16.4.1.5 Міжсайтовий скриптинг (XSS)

**XSS** [1556] — ін'єкція шкідливих скриптів (JavaScript) в легітимні сайти. SOP-засоби неефективні, оскільки скрипт надходить з оригінального хоста.

Типи:
- **Збережений XSS (Stored/Persistent/Type-I)**: шкідливий скрипт постійно зберігається на сервері.
- **Відображений XSS (Reflected/Non-persistent/Type-II)**: скрипт відображається сервером у відповідь (розповсюджується через посилання).

Пом'якшення: суворе підтвердження вводу користувача (whitelist); безпечне кодування сутностей; бібліотека безпечного кодування; статичний аналіз коду.

Дослідницький підхід: **рандомізація HTML-тегів/атрибутів** [1561] — щоб клієнти могли відрізняти довірений від ненадійного контенту.

#### 16.4.1.6 Підроблений міжсайтовий запит (CSRF)

**CSRF** [1557] — жертв змушують надіслати шкідливий HTTP-запит, успадковуючи їхню ідентичність та дозволи. Небезпечно для автентифікованих користувачів.

Неефективні контрзаходи: секретні cookie (всі cookie надсилаються), HTTPS (протокол не має значення), POST-запити (форми з прихованими полями).

Ефективне пом'якшення: **рандомізовані токени** в чутливих запитах (унікальні для сесії, згенеровані CSPRNG); сервер відхиляє запити без дійсного токена.

### 16.4.2 Неправильні конфігурації серверної сторони та вразливі компоненти

Веб-стек: веб-сервер, фреймворк застосунку, БД, брандмауер, балансувальник навантаження. Один небезпечний компонент може скомпрометувати весь стек.

**Heartbleed (2014)** [1558, 1559]: вразливість у OpenSSL — витік пам'яті веб-сервера (приватні ключі, паролі, номери кредитних карток).

**Брандмауер**: доступ лише до необхідних портів (80/443); конфігурація SSH лише з внутрішньої мережі.

**Балансувальники навантаження**: розподіл трафіку, контроль доступу через .htaccess, обмеження швидкості (rate limiting), пом'якшення DoS-атак, управління TLS.
