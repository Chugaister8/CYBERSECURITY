# Розділ 12. Безпека розподілених систем

**Neeraj Suri** — Університет Ланкастера

---

## ВСТУП

Розподілена система — це, як правило, сукупність географічно розосереджених ресурсів (обчислювальних та комунікаційних), що колективно надають сервіси, забезпечують висококоректний і надійний доступ до ресурсів через реплікацію та уможливлюють загальні можливості.

Безпека розподілених систем вирішує загрози, що виникають від використання вразливостей у поверхнях атаки, створених у структурі ресурсів та функціональних можливостей розподіленої системи. Це охоплює ризики для потоків даних, механізмів контролю доступу, механізмів транспортування даних, сервісів координації ресурсів (реплікація, обробка відмов, транзакційна обробка, узгодженість даних) та розподілених застосунків.

## ЗМІСТ

## 12.1 КЛАСИ РОЗПОДІЛЕНИХ СИСТЕМ ТА ВРАЗЛИВОСТІ
*[1056, c2][1057, c5][1058, c18]*

### 12.1.1 Класи розподілених систем

Розподілені системи можна широко класифікувати за схемою координації:

**Клас 1: Децентралізовані однорангові взаємодії без централізованого сервісу координації — P2P (Peer-to-Peer) системи.** Децентралізований неконтрольований за часом характер є їх ключовою характеристикою. Приклади: Kademlia, Napster, Gnutella.

**Клас 2: Координований кластеринг розподілених ресурсів та сервісів.** Підкласи:
- **(a) Координація ресурсів**: Хмарні системи, клієнт-серверні моделі, Великі Дані, HPC.
- **(b) Координація сервісів**: транзакційні бази даних, розподілені реєстри, системи зберігання.

Розподілена система — часто агрегація кількох рівнів: ресурси пристрою → ОС → розподілені сервіси (іменування, синхронізація часу, розподілені файлові системи) → Middleware (RPC, публікація-підписка, Enterprise Service Bus) → розподілені застосунки.

### 12.1.2 Класи вразливостей та загроз

**Контроль доступу/допуску та управління ідентичністю** — маскування/спуфінг ідентичності, DoS-атаки на доступність.

**Транспортування даних** — пасивні (підслуховування) або активні атаки (модифікація даних), MITM-атаки.

**Управління ресурсами та сервіси координації** — загрози синхронізації, управлінню реплікацією, зміні виду, впорядкованості подій, консенсусу, транзакційному підтвердженню.

**Безпека даних** — CIA (конфіденційність, цілісність, доступність) для всіх елементів ланцюжка даних.

## 12.2 ДЕЦЕНТРАЛІЗОВАНІ P2P-МОДЕЛІ
*[1062, c11-12][1057, c25]*

P2P-системи характеризуються масштабованістю, децентралізованою координацією та низькою вартістю. П'ять основних принципів P2P: симетрія інтерфейсів, стійкість до збоїв, виживаність даних/сервісів через реплікацію, використання ресурсів на «краю» мережі, мінливість надання ресурсів між вузлами.

Три базові операції P2P: (a) ідентифікація/іменування вузлів, (b) схеми маршрутизації між вузлами, (c) виявлення вузлів.

### 12.2.1 Неструктуровані P2P-протоколи

Приклади: Freenet, Gnutella [1064, 1065] — переважно для поширення даних. Топологія вузлів неструктурована (дерева або сітки). Ресурси шукаються за іменем/міткою без структурованого адресування (пошук в ширину, в глибину, випадкові прогулянки). Вузли зберігають таблиці маршрутизації з контактами інших вузлів і регулярно «пінгують» їх.

### 12.2.2 Структуровані P2P-протоколи

Приклади: Chord, Pastry, Tapestry, Kademlia, CAN [1066-1069] — для ефективного виявлення даних. Топологія — «тісний світ» (ring зі ярликами). Використовують **DHT (Distributed Hash Table)**: ресурси/дані зберігаються за ключем у розподіленій структурі даних. Адресний простір — ціле число в діапазоні [0, 2^w−1]. Механізм пошуку послідовно зменшує відстань до цільового ідентифікатора.

### 12.2.3 Гібридні P2P-протоколи

Поєднують неструктурований та структурований підходи. Приклади: Napster, BitTorrent [1070]. BitTorrent відмовився від «трекер-серверів», ставши повністю децентралізованим.

### 12.2.4 Ієрархічні P2P-протоколи

Деякі вузли беруть на себе координуючу роль («суперузли»). Приклади: eDonkey, KaZaA. Покращує продуктивність пошуку та зменшує трафік.

## 12.3 АТАКИ НА P2P-СИСТЕМИ
*[1058, c16][1073, c5]*

Функціональні елементи P2P, що потребують захисту:
- **P-OP** (P2P Operations): виявлення, запит, маршрутизація, завантаження.
- **P-DS** (P2P Data Structures): таблиці маршрутизації, спільні ресурси.

### 12.3.1 Типи атак

**Таблиця 12.1: Атаки P2P, цілі безпеки та уражені функції**

| Атака | Доступність | Цілісність | Конфіденційність | Функція |
|---|---|---|---|---|
| DoS/DDoS | ✓ | ✗ | ✗ | P-OP |
| Змова (Collusion) | ✓ | ✓ | ✓ | P-OP |
| Забруднення (Pollution) | ✗ | ✓ | ✗ | P-DS |
| Відмивання та цензура | ✓ | ✓ | ✗ | P-DS |
| Маршрутизація | ✓ | ✓ | ✗ | P-DS |
| Буфер-карта (Buffer map cheating) | ✓ | ✓ | ✗ | P-OP |
| Sybil | ✓ | ✗ | ✓ | P-OP |
| Eclipse | ✓ | ✓ | ✓ | P-DS, P-OP |

- **DoS/DDoS** [1074]: виснаження ресурсів; обмеження доступу до вузла або каналу зв'язку.
- **Атаки змови** [1078]: достатньо велика підмножина вузлів узгоджено проводить стратегію проти P2P-сервісів.
- **Атаки забруднення** [1079, 1080]: додавання некоректної інформації. Приклад: «тифозна рекламна атака» — частковий вміст змінюється та поширюється.
- **Відмивання/цензура** [1080]: незаконна зміна, видалення або відмова в доступі до даних. Небезпечно для систем репутації.
- **Атаки маршрутизації** [1077, 1082]: підрив механізму передачі повідомлень. **Отруєння таблиць маршрутизації (RTP)** — зловмисник навмисно змінює таблиці маршрутизації свої або сусідів.
- **Буфер-карта** [1083]: зловмисники зменшують вихідний трафік, брешучи про наявність даних (актуально для потокових медіа-застосунків P2P).
- **Sybil-атаки** [1084]: вставка у мережу множини вузлів, контрольованих одним зловмисником. Можуть порушити системи голосування або репутації.
- **Eclipse-атаки** [1085]: добросовісний вузол оточується змовницькою групою шкідливих вузлів, що повністю або частково блокує його бачення решти системи.

### 12.3.2 Атаки та їхнє пом'якшення

**Механізми безпеки P2P**: механізми аутентифікації, безпечне зберігання, безпечна маршрутизація.

**Sybil та Eclipse**: пом'якшення — централізований орган для реєстрації вузлів, сертифікати від CA. Криптографія з відкритим ключем для перешкоджання вибору своїх мережевих ID.

**Маршрутизація**: множинні шляхи для кожного пошуку через розділені шляхи (за рахунок великих накладних витрат на повідомлення).

## 12.4 КООРДИНОВАНИЙ КЛАСТЕРИНГ РЕСУРСІВ
*[1062, c5,7,12,25][1056, c3][1057, c5,c14][1058, c16-17,c19]*

На відміну від децентралізованого управління P2P, велика кількість розподілених систем використовує координаційні механізми, що надають ілюзію логічно централізованої та скоординованої системи.

### 12.4.1 Стилі координації систем

1. **Синхронний**: всі компоненти координовані у часі (lock-step). Типово для систем реального часу.
2. **Асинхронний**: окремі сутності діють у довільному порядку. Типово для транзакційних систем, баз даних.
3. **Частково синхронний**: деякі обмеження на впорядкованість, але без lock-step. Типово для систем SCADA.

### 12.4.2 Надійна та безпечна групова комунікація

Включає точка-точка обмін повідомленнями, надійний і безпечний мультикаст (атомний, best-effort, регулярний, рівномірний, з журналюванням, тощо), публікація-підписка. Шифрування та криптографічний підпис каналів і повідомлень збільшують накладні витрати.

### 12.4.3 Властивості координації

**Консенсус** — досягнення угоди щодо значень. Властивості: узгодженість (всі коректні процеси погоджуються), дійсність (погоджене значення є дійсним), завершення (рішення в кінцевому підсумку досягається).

**Членство та узгодженість:**

- **Сильна узгодженість**: учасники мають погодитися на одному узгодженому порядку дій.
  - *Строга узгодженість*: немає обмежень на спостережуваний порядок, якщо він узгоджений між усіма.
  - *Лінеаризація (Linearisability)*: строга узгодженість з додатковим обмеженням, що порядок відповідає реальному часу.
- **Слабка узгодженість**: учасники не обов'язково спостерігають однаковий порядок.
  - *Послідовна узгодженість*: порядок виконання кожного процесу зберігається.
  - *Причинна узгодженість*: лише порядок причинно пов'язаних дій зберігається.
  - *Остаточна узгодженість*: учасники рано чи пізно сходяться до узгодженого стану.

Системи зі слабкою узгодженістю стали популярними завдяки вищій доступності (Amazon Dynamo, Facebook Cassandra).

### 12.4.4 Управління реплікацією та схеми координації

**Стейт-машинна реплікація** [1103] — загальний метод реалізації відмовостійкого сервісу через реплікацію серверів і координацію взаємодій клієнтів з репліками.

**Теорема CAP**: будь-яка мережева система з розподіленими даними може забезпечити лише 2 з 3 властивостей [1104]:
1. **C (Consistency)**: еквівалентно наявності єдиної актуальної копії даних.
2. **A (Availability)**: кожен запит в кінцевому підсумку отримує відповідь.
3. **P (Partition tolerance)**: сервери не можуть бути розбиті на групи без зв'язку.

**Paxos** [1105]: група протоколів неявного вибору лідера для вирішення консенсусу в асинхронному середовищі. Tolerant до збоїв (потрібна більшість). Реалізований у Google Chubby [1106]. Відомий складністю реалізації; **RAFT** [1107] — більш простий аналог.

**Візантійська відмовостійкість (BFT)** [1108]: гарантує коректне виконання операцій, якщо не більше 1/3 процесів скомпрометовано та поводяться довільно (візантійськи). **FLP-результат про неможливість** [1109]: неможливо досягти консенсусу при асинхронній комунікації. Потребує >3f вузлів для обробки f шкідливих відмов.

**Двофазне підтвердження (2PC)**: координатор запитує всіх клієнтів підтвердити або скасувати транзакцію. Широко використовується. Класичний 2PC вразливий до збоїв координатора.

**Трифазне підтвердження (3PC)**: розширення 2PC, більш робастне, але менш використовується через витрати на обмін повідомленнями.

## 12.5 КЛАСИ КООРДИНАЦІЇ ТА ВРАЗЛИВІСТЬ
*[1062, c3][1056, c5,c6][1057, c19][1058, c18][1073, c3]*

### Загальний клас порушень

Атаки у розподілених системах можна класифікувати за рівнем доставки повідомлень:

1. **Засновані на часі**: пропуски повідомлень, ранні, запізнілі або позачергові повідомлення. Збої та DoS.
2. **Засновані на значеннях/інформації**: спуфінг, мімікрія, дублювання, витік інформації (бічні канали, приховані канали), маніпуляція вмістом. **Візантійська атака** — навмисно модульована інформація різним групам ресурсів → несумісні погляди на стан системи.

### 12.5.1 Клас координації ресурсів — інфраструктурний погляд

**Хмарна модель**: «ресурсна платформа» для виконання сервісів. ENISA, NIST та ISO специфікації IaaS та PaaS. Поверхня атаки Хмари: ресурси, мережа, VM, планувальники, брокери, моніторинг.

Сценарії атак (та пом'якшення):

- **Компрометація ресурсів**: впливає на доступність. *Пом'якшення*: контроль доступу, брандмауери, авторизація, пісочниця, надійна обчислювальна база (TCB).
- **Компрометація контролю доступу/допуску** (маскування, спуфінг, управління ідентичністю): впливає на доступність, цілісність та конфіденційність. *Пом'якшення*: IDS, регулярні запити аутентифікації.
- **Компрометація VM** (витік через бічні або приховані канали): порушення цілісності та конфіденційності. *Пом'якшення*: аналіз заражень (taint analysis), апаратні лічильники продуктивності [1121], гіпервізори.
- **Компрометація планувальника**: впливає на доступність та цілісність. *Пом'якшення*: контроль доступу та координаційні конструкції.
- **Компрометація брокера**: впливає на доступність ресурсів. *Пом'якшення*: резервні брокери.
- **Компрометація комунікацій**: фундаментально порушує функціональність системи. *Пом'якшення*: різноманітні техніки захисту комунікацій (повтори, ACK/NACK, криптографічні канали).
- **Компрометація моніторингу та обліку**: *Пом'якшення*: схеми узгодженості стану.

### 12.5.2 Клас координації сервісів — погляд на застосунки

**Веб-сервіси**: видобуток даних, веб-краулери, інформаційні сервери, підтримка e-транзакцій.

**Розподіл ключів** (Kerberos, PKI): аутентифікація у нешифрованих мережах через криптографічні протоколи.

**Зберігання/KVS**: від реєстрового рівня до Key Value Store. Безпекові питання стосуються даних у стані спокою (static storage) та у русі (динамічні операції R/W).

**Транзакційні сервіси, бази даних**: вимоги ACID (атомарність, узгодженість, ізольованість, довговічність). Компрометація: порушення цілісності транзакцій.

**Блокчейни/Криптовалюти**: децентралізований розподілений реєстр — список блоків, де кожен блок включає криптографічний хеш попереднього блоку та мітку часу. Забезпечує незмінність попередніх даних та збереження порядку транзакцій.

**Дозвільний (permissioned)** vs **бездозвільний (permission-less)** блокчейн.

**Proof-of-Work (PoW)**: обчислювально дорогі задачі для запобігання зловживанню сервісами. Недолік: велике споживання енергії. **Proof-of-Stake (PoS)** — альтернатива у розробці.

Компрометація Блокчейну: підробка криптографічних ключів і хешів, **атака 51%** (>50% обчислювальної потужності), **Eclipse-атака** [1124] (обмеження комунікацій вузла та підміна даних).

**Атаки на сервіси** та пом'якшення аналогічні описаним у Розділі 12.5.1: реплікаційне управління та координаційні конструкції є основою підходів до пом'якшення.

## ВИСНОВКИ

Розуміння концептуальних основ розподілених систем є фундаментальним для розуміння та забезпечення їхньої безпеки. Ця ГЗ розглянула функціональну категоризацію розподілених систем на два великі класи: децентралізований та координований контроль. Операції для кожного класу були проаналізовані з виявленням відповідних наслідків для безпеки. Схеми реплікаційного управління та координації є основою підходів до пом'якшення загроз.
