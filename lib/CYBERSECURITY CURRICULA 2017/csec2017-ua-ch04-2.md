# 4.2 Галузь знань: Безпека програмного забезпечення

Галузь знань «Безпека програмного забезпечення» зосереджена на розробленні та використанні програмного забезпечення, яке надійно зберігає властивості безпеки інформації та систем, що воно захищає. Безпека системи, а також даних, які вона зберігає та якими керує, значною мірою залежить від безпеки її програмного забезпечення. Безпека програмного забезпечення залежить від того, наскільки вимоги відповідають потребам, які воно має задовольняти, наскільки добре програмне забезпечення спроектовано, реалізовано, протестовано, розгорнуто та підтримується. Документація є критично важливою для розуміння цих міркувань, а етичні питання виникають упродовж усього циклу створення, розгортання, використання та виведення з експлуатації програмного забезпечення.

Галузь знань «Безпека програмного забезпечення» розглядає ці проблеми безпеки. Одиниці знань у межах цієї галузі знань складаються з фундаментальних принципів і практик.

## 4.2.1 Одиниці знань та теми

Наведена нижче таблиця перелічує основні елементи, одиниці знань та теми галузі знань «Безпека програмного забезпечення». Ці одиниці знань були перевірені Робочою групою з безпеки програмного забезпечення з використанням OWASP Top 10 та публікації IEEE «Avoiding the Top 10 Software Security Design Flaws».

---

**БЕЗПЕКА ПРОГРАМНОГО ЗАБЕЗПЕЧЕННЯ**

**Основні елементи:**
- Фундаментальні принципи проектування, включаючи найменші привілеї, відкритий дизайн та абстракцію,
- Вимоги безпеки та їхня роль у проектуванні,
- Проблеми реалізації,
- Статичне та динамічне тестування,
- Конфігурування та встановлення виправлень,
- Етика, особливо при розробленні, тестуванні та розкритті вразливостей.

---

| Одиниці знань | Теми | Опис / Навчальні рекомендації |
|---|---|---|
| **Фундаментальні принципи** *[Див. також KA «Безпека компонентів» щодо пов'язаного змісту, с. 29.]* | | Ця одиниця знань знайомить із принципами, що лежать в основі як проектування, так і реалізації. Перші п'ять є принципами обмеженості, наступні три — принципами простоти, а решта — принципами методології. |
| | Найменші привілеї | Програмному забезпеченню слід надавати лише ті привілеї, які необхідні для виконання його завдання. |
| | Безпечні значення за замовчуванням | Початковий стан повинен передбачати заборону доступу, якщо тільки доступ явно не потрібен. Якщо програмному забезпеченню не надано явного доступу до об'єкта, у доступі до цього об'єкта йому слід відмовити, а стан захисту системи повинен залишатися незмінним. |
| | Повне посередництво | Програмне забезпечення повинно перевіряти кожен доступ до об'єктів, щоб переконатися, що доступ дозволено. |
| | Розмежування | Програмне забезпечення не повинно надавати доступ до ресурсу або виконувати дію, що стосується безпеки, на основі однієї умови. |
| | Мінімізація довіри | Програмне забезпечення повинно перевіряти всі вхідні дані та результати всіх дій, що стосуються безпеки. |
| | Економія механізму | Функції безпеки програмного забезпечення повинні бути якомога простішими. |
| | Мінімізація спільного механізму | Спільне використання ресурсів слід зводити до мінімуму. |
| | Найменше здивування | Функції безпеки програмного забезпечення та механізми безпеки, які воно реалізує, повинні бути спроектовані так, щоб їхня робота була якомога логічнішою та простішою. |
| | Відкритий дизайн | Безпека програмного забезпечення та того, що воно забезпечує, не повинна залежати від таємності його дизайну чи реалізації. |
| | Багаторівневість | Організовуйте програмне забезпечення шарами так, щоб модулі на певному шарі взаємодіяли лише з модулями безпосередньо вище і нижче. Це дозволяє тестувати програмне забезпечення по одному шару за раз, використовуючи техніки «зверху вниз» або «знизу вгору», і скорочує точки доступу, дотримуючись принципу розмежування. |
| | Абстракція | Приховуйте внутрішню будову кожного шару, роблячи доступними лише інтерфейси; це дозволяє змінювати спосіб виконання завдань шаром, не зачіпаючи компоненти на інших шарах. |
| | Модульність | Проектуйте та реалізуйте програмне забезпечення як сукупність взаємодіючих компонентів (модулів); при цьому кожен інтерфейс модуля є абстракцією. |
| | Повна зв'язність | Пов'язуйте дизайн і реалізацію безпеки програмного забезпечення зі специфікаціями безпеки для цього програмного забезпечення. |
| | Проектування для ітерацій | Плануйте дизайн таким чином, щоб його можна було змінити за потреби. Це мінімізує наслідки для безпеки при зміні дизайну, якщо специфікації не відповідають середовищу, в якому використовується програмне забезпечення. |
| **Проектування** | | Ця одиниця знань описує техніки включення міркувань безпеки на всіх етапах проектування програмного забезпечення. |
| | Виведення вимог безпеки | Починаючи з бізнесових, місійних або інших цілей, визначте, які вимоги безпеки необхідні для їх досягнення. Вони також можуть бути виведені або змінені в міру розвитку програмного забезпечення. |
| | Специфікація вимог безпеки | Перетворіть вимоги безпеки у форму, придатну для використання (формальна специфікація, неформальні специфікації, специфікації для тестування). |
| | Життєвий цикл розроблення програмного забезпечення / Lifecycle безпечного розроблення | Включає такі приклади: каскадна модель, гнучка розробка та безпека. |
| | Мови програмування та типобезпечні мови | Розглядаються проблеми, які вносять мови програмування, що таке типобезпека та чому вона важлива. |
| **Реалізація** | | Ця одиниця знань описує техніки включення міркувань безпеки на всіх етапах реалізації програмного забезпечення. |
| | Перевірка вхідних даних та їхнього представлення | Перевіряйте межі буферів і значення цілих чисел, щоб переконатися, що вони знаходяться в допустимому діапазоні; перевіряйте вхідні дані, щоб переконатися, що вони відповідають очікуванням і будуть оброблені/інтерпретовані коректно. |
| | Коректне використання API | Переконайтеся, що параметри та середовища перевірені та контрольовані так, щоб API правильно застосовував політику безпеки; перевіряйте результати використання API на наявність проблем. |
| | Використання функцій безпеки | Використовуйте криптографічну випадковість; належним чином обмежуйте привілеї процесів. |
| | Перевірка відносин часу та стану | Перевіряйте, що файл, з яким виконуються дії, — це той файл, для якого перевірялися відповідні атрибути; перевіряйте, що процеси виконуються. |
| | Належна обробка винятків та помилок | За необхідності блокуйте або ставте в чергу сигнали під час їхньої обробки; визначайте, яку інформацію слід надавати користувачеві, балансуючи між зручністю використання та необхідністю приховати деяку інформацію, а також як і кому повідомляти цю інформацію. |
| | Надійне програмування | Ця тема іноді називається захищеним або захисним програмуванням. Навчальний зміст повинен включати: звільняйте лише виділену пам'ять; ініціалізуйте змінні перед використанням; не покладайтеся на невизначену поведінку. |
| | Інкапсуляція структур і модулів | Ця тема включає класи та інші інстанціації. Приклад: ізоляція процесів. |
| | Врахування середовища | Приклад: не розміщуйте конфіденційну інформацію у вихідному коді. |
| **Аналіз та тестування** *[Див. також KA «Безпека компонентів» щодо пов'язаного змісту, с. 29.]* | | Ця одиниця знань знайомить із міркуваннями щодо тестування для перевірки відповідності програмного забезпечення заявленим (та незаявленим) вимогам і специфікаціям безпеки. Незаявлені вимоги включають ті, що стосуються надійності загалом. |
| | Статичний та динамічний аналіз | Ця тема описує різні методи для кожного з них, включає те, як статичний і динамічний аналіз доповнюють один одного, а також обмеження та переваги кожного, а також те, як виконувати ці типи аналізу для дуже великих програмних систем. |
| | Модульне тестування | Ця тема описує, як тестувати складові частини програмного забезпечення, такі як модулі. |
| | Інтеграційне тестування | Ця тема описує, як тестувати компоненти програмного забезпечення в процесі їхньої інтеграції. |
| | Тестування програмного забезпечення | Ця тема описує, як тестувати програмне забезпечення загалом, і розміщує модульне та інтеграційне тестування у відповідній системі координат. |
| **Розгортання та обслуговування** | | Ця одиниця знань розглядає міркування безпеки при використанні програмного забезпечення, а також при його розгортанні, обслуговуванні та виведенні з експлуатації. |
| | Конфігурування | Ця тема охоплює налаштування програмної системи для забезпечення її коректної роботи. |
| | Встановлення виправлень та lifecycle вразливостей | Ця тема включає управління звітами про вразливості, усунення вразливостей, тестування виправлення та розповсюдження виправлень. |
| | Перевірка середовища | Ця тема охоплює перевірку відповідності середовища припущенням, закладеним у програмному забезпеченні, і, якщо ні, — способи вирішення конфлікту. |
| | DevOps | Ця тема поєднує розроблення та експлуатацію, а також автоматизацію та моніторинг обох. |
| | Виведення з експлуатації / Завершення роботи | Ця тема описує, що відбувається при видаленні програмного забезпечення, і як видалити його без виникнення проблем безпеки. |
| **Документація** | | Ця одиниця знань описує, як вводити та включати інформацію про міркування безпеки в конфігурацію, використання та інші аспекти застосування програмного забезпечення та його підтримки (включаючи модифікацію за потреби). |
| | Документи встановлення | Ця тема включає документацію з встановлення та конфігурування. |
| | Посібники та керівництва користувача | Ця тема включає навчальні посібники та шпаргалки (короткі довідники); вони повинні підкреслювати будь-які потенційні проблеми безпеки, які можуть спричинити користувачі. |
| | Документація із забезпечення якості | Ця тема зосереджена на тому, як була встановлена коректність і що означає коректність у цьому контексті. |
| | Документація з безпеки | Ця тема зосереджена на потенційних проблемах безпеки, способах їх уникнення, а якщо вони виникають — на можливих наслідках і способах їх вирішення. |
| **Етика** *[Див. також KA «Організаційна безпека», с. 51, та KA «Суспільна безпека», с. 62, щодо пов'язаного змісту.]* | | Ця одиниця знань знайомить із етичними міркуваннями в усіх перелічених вище сферах, щоб студенти могли міркувати про наслідки рішень та ефектів, пов'язаних із безпекою. |
| | Етичні питання в розробленні програмного забезпечення | Ця тема охоплює повторне використання коду (ліцензування), професійну відповідальність, кодекси етики, такі як ACM/IEEE-CS Software Engineering Code of Ethics and Professional Practice. |
| | Соціальні аспекти розроблення програмного забезпечення | Ця тема охоплює розгляд наслідків програмного забезпечення, що розробляється, — як при коректній роботі, так і при застосуванні неякісних або небезпечних практик програмування. |
| | Правові аспекти розроблення програмного забезпечення | Ця тема розглядає аспекти відповідальності за програмне забезпечення, нормативне регулювання, а також дотримання вимог і пов'язані з цим питання. |
| | Розкриття вразливостей | Ця тема охоплює: як розкривати, кому розкривати і коли розкривати («відповідальне розкриття»). |
| | Що, коли і навіщо тестувати | Ця тема описує етичні наслідки тестування, особливо включаючи граничні випадки. |

## 4.2.2 Основні елементи та результати навчання

Від студентів вимагається демонструвати компетентність у кожному з основних понять через досягнення результатів навчання. Як правило, результати навчання лежать у межах рівнів розуміння та застосування за переглянутою таксономією Блума (http://ccecc.acm.org/assessment/blooms).

| Основні елементи | Результати навчання |
|---|---|
| Фундаментальні принципи проектування; найменші привілеї, відкритий дизайн та абстракція | Обговорити наслідки покладання на відкритий дизайн або на таємність дизайну для безпеки. |
| | Перелічити три принципи безпеки. |
| | Описати, чому кожен принцип є важливим для безпеки. |
| | Визначити необхідний принцип проектування. |
| Вимоги безпеки та їхня роль у проектуванні | Пояснити, чому вимоги безпеки є важливими. |
| | Визначити поширені вектори атак. |
| | Описати важливість написання захищених і надійних програм. |
| | Описати концепцію конфіденційності, включаючи персональну ідентифікаційну інформацію. |
| Проблеми реалізації | Пояснити, чому перевірка вхідних даних та санітизація даних є необхідними. |
| | Пояснити різницю між псевдовипадковими числами та випадковими числами. |
| | Розмежувати захищене кодування та встановлення виправлень і пояснити перевагу використання технік захищеного кодування. |
| | Описати переповнення буфера та пояснити, чому це є потенційною проблемою безпеки. |
| Статичний, динамічний аналіз | Пояснити різницю між статичним та динамічним аналізом. |
| | Обговорити проблему, яку статичний аналіз не може виявити. |
| | Обговорити проблему, яку динамічний аналіз не може виявити. |
| Конфігурування, встановлення виправлень | Обговорити необхідність оновлення програмного забезпечення для усунення вразливостей безпеки. |
| | Пояснити необхідність тестування програмного забезпечення після оновлення, але до розповсюдження виправлення. |
| | Пояснити важливість коректного конфігурування програмного забезпечення. |
| Етика, особливо при розробленні, тестуванні та розкритті вразливостей | Пояснити концепцію того, що можливість зробити щось не означає, що це слід робити. |
| | Обговорити етичні питання при розкритті вразливостей. |
| | Обговорити етику ретельного тестування, особливо граничних випадків. |
| | Визначити етичні наслідки та вплив проектних рішень. |
