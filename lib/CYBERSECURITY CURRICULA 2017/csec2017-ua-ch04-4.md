# 4.4 Галузь знань: Безпека з'єднань

Галузь знань «Безпека з'єднань» зосереджена на безпеці з'єднань між компонентами, включаючи як фізичні, так і логічні з'єднання.

Для кожного фахівця з кібербезпеки є критично важливим мати базові знання з цифрових комунікацій та мережевих технологій. З'єднання — це спосіб взаємодії компонентів. Значна частина цього матеріалу може вводитися через приклади з подальшим узагальненням до основних елементів і введенням відповідної термінології. Разом із KA «Безпека компонентів» та KA «Безпека систем», KA «Безпека з'єднань» розглядає проблеми безпеки при підключенні компонентів та їхньому використанні в складі більших систем.

## 4.4.1 Одиниці знань та теми

Наведена нижче таблиця перелічує основні елементи, одиниці знань та теми галузі знань «Безпека з'єднань».

---

**БЕЗПЕКА З'ЄДНАНЬ**

**Основні елементи:**
- Системи, архітектура, моделі та стандарти,
- Фізичні інтерфейси компонентів,
- Програмні інтерфейси компонентів,
- Атаки на з'єднання,
- Атаки на передавання даних.

Різні матеріали мають різні характеристики та можливості передавання сигналу. Навіть нетехнічним фахівцям з безпеки необхідно розуміти, що оптичне волокно відрізняється від кручений пари і що кожне з них має різні стандарти та специфічні стандартні конектори. |

Як приклад може використовуватися історія материнських плат ПК, що показує еволюцію від ISA через PCI і далі. Важливою є здатність плат розширення додавати функціональність без зміни базової архітектури. Додавання багатопортових мережевих плат Ethernet дозволяє ПК стати маршрутизатором. |

| | Поширені архітектури | Ця тема повинна розглядати поточні технології, з якими зустрінуться студенти (чіпи CPU, материнська плата ПК, стандарти Ethernet). |
| **Архітектура розподілених систем** | | Ця одиниця знань знайомить із загальними концепціями розподілених систем та способами їхнього з'єднання. Інтернет — не єдина мережа, а TCP/IP — не єдиний протокол для з'єднання систем. Кожна реалізація має специфічні характеристики та різні потенційні вразливості. Основна увага в навчальній програмі повинна приділятися подібностям, відмінностям та причинам прийняття проектних рішень. Кожна архітектура має переваги та недоліки для конкретних випадків використання, а також особливі вразливості та сильні сторони з точки зору безпеки. Не можна припускати, що стратегія пом'якшення для Інтернету буде доцільною для інфраструктури суперкомп'ютера. |
| | Загальні концепції | Ця тема повинна починатися з ідеї процесу в операційній системі, а потім знайомити з різними архітектурами для запуску процесів та забезпечення їхньої комунікації. Симетрична багатопроцесорна обробка та спільна пам'ять, мережева модель з моделлю міжпроцесної комунікації. |
| | Всесвітня павутина | Ця тема охоплює протокол HTTP/HTTPS і демонструє, як він є прикладом стандарту розподіленої обробки даних. |
| | Інтернет | Ця тема охоплює еволюцію Інтернету як платформи розподіленої обробки даних. Студенти повинні чітко розуміти, чому Всесвітня павутина та Інтернет не є еквівалентними поняттями. |
| | Протоколи та багаторівневість | Ця тема охоплює семирівневу модель OSI разом із п'ятирівневою моделлю Інтернету та порівнює їх як приклад інкапсуляції та багаторівневості для забезпечення послуг, що базуються одна на одній. |
| | Високопродуктивні обчислення (суперкомп'ютери) | Ця тема знайомить із HPC та випадками використання, що відрізняють HPC від стандартного Інтернету. |
| | Гіпервізори та реалізації хмарних обчислень | Ця тема знайомить із концепціями надання інфраструктури як послуги (IaaS), програмного забезпечення як послуги (SaaS), платформи як послуги (PaaS) та всіх їхніх похідних, релевантних для студентів. |
| | Вразливості та приклади експлойтів | Ця тема розглядає поверхні атак різних моделей розподілених обчислень, підкреслюючи той факт, що кожен інтерфейс вносить потенційні вразливості. Гіпервізор, віртуальна мережа, фізична мережа та міжпроцесна комунікація повинні бути охоплені. |
| **Мережева архітектура** *[Див. також KA «Безпека систем» щодо пов'язаного змісту, с. 39.]* | | Ця одиниця знань знайомить із концепціями, що зазвичай охоплюються в курсі комп'ютерних мереж. Вона надає основу для більш спеціалізованих одиниць знань. |
| | Загальні концепції | Ця тема повинна охоплювати ідеї вузлів і ребер з назвами різних топологій та характеристиками передавання топологій. |
| | Поширені архітектури | Ця тема охоплює мережеву архітектуру IEEE 802 та те, як різні мережі називаються за фізичними характеристиками (LAN, MAN тощо). |
| | Пересилання | Ця тема охоплює пересилання пакетів загалом. Оскільки схожа комутаційна мікросхема тепер використовується в маршрутизаторах і комутаторах, а SDN розглядає пересилання окремо від побудови таблиці пересилання, це є окремою темою. |
| | Маршрутизація | Ця тема охоплює алгоритми маршрутизації та пояснює, як таблиці пересилання будуються з використанням алгоритмів аналізу графів, таких як стан каналу та вектор відстані. |
| | Комутація / Міст | Ця тема охоплює алгоритми навчання та мостування IEEE 802.1 разом із протоколом Spanning Tree та його зв'язком із маршрутизацією. |
| | Нові тенденції | Ця тема охоплює нові технології та їхній вплив у міру їхньої появи. Наразі контентом є вплив SDN та додавання маршрутизації до рівня 2 з удосконаленими навчальними мостами. Це швидко розвивається. |

| **Мережеві реалізації** | | Ця одиниця знань досліджує конкретні технології, що реалізують загальні концепції мережевих технологій. Концепції мережевої архітектури можуть ілюструватися конкретними реалізаціями, але слід чітко зазначити, що існують інші можливості. Слід підкреслити, що вразливості експлуатуються в реалізаціях. Часто архітектура може бути теоретично перевірена як коректна, але реалізована у спосіб, що має вразливості. Також стики між технологіями часто відкривають вразливості. ARP-спуфінг є прикладом того, як стик між технологіями відкриває вразливості. |
| | Мережі IEEE 802/ISO | Ця тема є глибоким занурення в стандарти ISO. Очікується, що ця тема буде представлена в інших місцях. |
| | Мережі IETF та TCP/IP | Це глибоке занурення в базову інфраструктуру Інтернету та TCP. |
| | Практичні протоколи інтеграції та «склеювання» | Ця тема розглядає проблему інтеграції технологій через реалізацію того, що можна назвати інтерфейсними прошарками або «клейовим» кодом. ARP є очевидним прикладом. Був необхідний механізм для відображення IP-адрес моделі міжмережевої взаємодії IETF на MAC-адреси базових мереж. ARP — це «клей». Аналогічно, Infiniband потребує прошарку для передавання IP-трафіку. |
| | Вразливості та приклади експлойтів | Ця тема повинна наводити приклади з технологій, важливих для програми. Якщо ARP обирається як приклад, ARP-спуфінг як атака «людина посередині» є гарним прикладом. USB та інші послідовні з'єднання також можуть слугувати прикладами. |
| **Мережеві сервіси** | | Ця одиниця знань досліджує різні моделі, що використовуються для реалізації з'єднання між споживачем послуги та її постачальником. Кожна тема може розглядатися на багатьох рівнях із багатьма прикладами (наприклад, бездротові проблеми, пов'язані з біомедичними пристроями). Цю область виділено окремо, оскільки сервісні моделі можуть бути реалізовані у стільки різних способів і з такою кількістю різних архітектур. Виклики віддалених процедур (RPC) реалізуються через багато різних технологій з'єднання: від процес-до-процесу в одному процесорі до з'єднань через Інтернет. |
| | Концепція сервісу | Ця тема є мережево-орієнтованим зануренням в одну модель розподілених обчислень. Сервіс — це процес, який надає щось іншому процесу на основі запиту. |
| | Сервісні моделі (клієнт-сервер, рівний-до-рівного) | Ця тема є мережево-орієнтованим поглядом на те, як моделюються сервіси. З мережевої точки зору клієнт ініціює з'єднання, а сервер відповідає. У P2P будь-яка сторона може ініціювати запит. |
| | Концепції сервісних протоколів (IPC, API, IDL) | Ця тема описує всі способи з'єднання компонентів: виклики процедур, IPC-запити, мови визначення інтерфейсів зі stub-кодом, власні протоколи через сокет тощо. |
| | Поширені архітектури сервісних комунікацій | Ця тема розглядає конкретні сервіси та реалізацію їхніх протоколів. Приклади: SMTP, HTTP, SNMP, REST, CORBA тощо. Також можуть розглядатися спеціалізовані з'єднання, такі як бездротове керування імплантованими медичними пристроями. |
| | Віртуалізація сервісів | Ця тема охоплює віртуалізацію сервісів як метод емуляції поведінки конкретних компонентів, таких як хмарні застосунки та сервісно-орієнтована архітектура. |
| | Вразливості та приклади експлойтів | Ця тема розглядає вразливості та експлойти мережевих сервісів клієнт-сервер, рівний-до-рівного та з використанням віртуалізації. Поширені сервісні сигнатури часто використовуються для профілювання вразливостей. |
| **Мережевий захист** | | Ця одиниця знань охоплює поточні концепції захисту мережі. Ймовірно, що термінологія та технологія з часом суттєво зміняться. Ключові ідеї повинні включати вразливості з'єднань, такі як впровадження відгалуження в конектор та увімкнення підслуховування. Усе це надає вразливості, що можуть бути використані для атак «людина посередині». Необхідно охопити ідею базового захоплення та моніторингу відхилень від базового рівня, оскільки це застосовується в кількох конкретних темах. |
| | Зміцнення мережі | Ця тема охоплює способи допомогти мережі захиститися від несанкціонованого доступу. |
| | Впровадження IDS/IPS | Ця тема охоплює сервіси виявлення та запобігання вторгненням. Ці сервіси виконують аудит мережевого трафіку. |
| | Впровадження міжмережевих екранів та VPN | Ця тема охоплює встановлення та використання міжмережевих екранів та віртуальних приватних мереж. |
| | Ешелонований захист | Ця тема знайомить із ідеєю про те, що захист повинен бути багаторівневим. |
| | Honeypot-и та honeynets | Ця тема знайомить із ідеєю надання навмисно вразливих мереж і пристроїв в ізольованих мережах, щоб за ними можна було спостерігати та аналізувати в процесі атаки. |
| | Моніторинг мережі | Ця тема охоплює інструменти та техніки для моніторингу мережевих пристроїв та їхніх відповідних журналів. |

Приклади включають техніки мережевого контролю допуску; машинні сертифікати; техніки профілювання машин; зондування через SNMP, DHCP, HTTP, DNS, LDAP та NMAP. |
| | Демілітаризовані зони (DMZ) / Проксі-сервери | Ця тема охоплює інструменти та техніки реалізації ешелонованого захисту з використанням ізольованих мереж та спеціальних серверів. |
| | Розроблення та застосування мережевої політики | Ця тема охоплює створення політик, що надають рекомендації та вимоги щодо сервісів мережі, разом із заходами для забезпечення дотримання цих політик. |
| | Мережеві операційні процедури | Ця тема розглядає створення процедур, що використовуються для управління мережею. |
| | Мережеві атаки (наприклад, захоплення сесії, атака «людина посередині») | Ця тема охоплює інструменти та техніки тестування мережі через фактичні спроби експлуатації вразливостей. |
| | Полювання на загрози та машинне навчання | Ця тема охоплює те, як проактивне полювання на загрози використовує машинне навчання для виявлення шаблонів у векторах атак. |

## 4.4.2 Основні елементи та результати навчання

Від студентів вимагається демонструвати компетентність у кожному з основних понять через досягнення результатів навчання. Як правило, результати навчання лежать у межах рівнів розуміння та застосування за переглянутою таксономією Блума (http://ccecc.acm.org/assessment/blooms).

| Основні елементи | Результати навчання |
|---|---|
| Системи, архітектура, моделі та стандарти | Обговорити необхідність спільних моделей та архітектур для опису систем. |
| | Описати модель систем, що складається з компонентів та інтерфейсів для з'єднань. |
| | Пояснити, чому компонент потребує принаймні одного інтерфейсу. |
| | Перелічити кілька стандартів, що визначають моделі, які складаються з систем компонентів та інтерфейсів. |
| | Описати компоненти та інтерфейси наданого мережевого стандарту. |
| Фізичні інтерфейси компонентів | Пояснити, чому апаратний пристрій завжди моделюється як фізичний компонент. |
| | Навести кілька прикладів фізичних інтерфейсів компонентів із відповідними вразливостями. |
| | Описати експлойт для наданої вразливості фізичного інтерфейсу. |
| Програмні інтерфейси компонентів | Пояснити, чому кожен фізичний інтерфейс має відповідний програмний компонент для забезпечення відповідного програмного інтерфейсу. |
| | Пояснити, як програмні компоненти організовані для представлення логічних рівнів у стандартній моделі. |
| | Обговорити, як п'ятирівнева модель Інтернету може розглядатися як програмні компоненти та інтерфейси, що представляють рівні сервісів, інкапсульованих сервісами нижчого рівня. |
| | Обговорити, як TCP/IP як сервіс представлений різними інтерфейсами в різних програмних системах. |
| Атаки на з'єднання | Пояснити, як атаки на з'єднання можуть розглядатися як атаки на програмні інтерфейси компонентів. |
| | Описати, як наданий стандартний інтерфейс може відкривати вразливості в програмному компоненті, що реалізує цей інтерфейс. |
| | Описати, як реалізація може захистити себе від наданої вразливості у наданому стандартному інтерфейсі. |
| Атаки на передавання даних *[Див. також KA «Безпека даних» щодо пов'язаного змісту, с. 16.]* | Пояснити, як атаки на передавання часто реалізуються як атаки на компоненти, що надають сервіс ретрансляції інформації. |
| | Описати атаку на заданий вузол у мережі TCP/IP, враховуючи опис вразливості. |
| | Пояснити, чому атаки на передавання часто можна розглядати як атаки на з'єднання між мережевими компонентами (фізичними або програмними). |
